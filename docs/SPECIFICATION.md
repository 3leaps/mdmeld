# MDMeld Format Specification v1.0

## Overview

MDMeld is a text-based archive format that encodes a directory tree into a single markdown file. It is designed for sharing codebases with AI assistants through web chat interfaces, where file upload may be limited to text.

An MDMeld archive is valid markdown. Any markdown viewer can render it. The structured YAML frontmatter enables programmatic extraction of individual files without reading the entire archive.

## Archive Structure

An archive consists of three sections in order:

1. **Header comment** — an HTML comment identifying the file as an MDMeld archive
2. **YAML frontmatter** — file metadata and position indexes between `---` delimiters
3. **Content section** — file contents in fenced code blocks

### 1. Header Comment

```markdown
<!-- MDMeld v1.0 archive — a directory tree encoded as markdown.
     Read the manifest below for file listing and positions.
     Spec: https://github.com/3leaps/mdmeld -->
```

The header comment is optional when parsing but always generated by conforming packers. It allows humans and AI models encountering the format for the first time to understand what they are reading without needing external documentation.

### 2. YAML Frontmatter

The frontmatter is enclosed between `---` delimiters and contains two top-level keys:

#### `mdmeld` — Archive Metadata

| Field            | Type    | Required | Description                                    |
| ---------------- | ------- | -------- | ---------------------------------------------- |
| `version`        | string  | yes      | Format version (`"1.0.0"`)                     |
| `created`        | string  | no       | ISO 8601 timestamp                             |
| `hash_algorithm` | string  | yes      | `"xxh64"` or `"sha256"`                        |
| `backtick_count` | integer | yes      | Number of backticks used for code fences (4–8) |
| `files`          | array   | yes      | File entry list (see below)                    |

#### File Entry

| Field      | Type    | Required | Description                                                            |
| ---------- | ------- | -------- | ---------------------------------------------------------------------- |
| `path`     | string  | yes      | POSIX relative path (forward slashes, no leading `./`)                 |
| `type`     | string  | yes      | `"text"` or `"binary"`                                                 |
| `size`     | integer | yes      | Original file size in bytes                                            |
| `hash`     | string  | yes      | Hash of the original file content (hex string)                         |
| `syntax`   | string  | no       | Syntax highlighting language hint (e.g., `"typescript"`, `"markdown"`) |
| `encoding` | string  | no       | Content encoding. Present only for binary files (`"base64"`)           |
| `position` | object  | yes      | Line-based position metadata (see below)                               |

#### Position Metadata

All values are 1-based line numbers within the archive.

| Field     | Type    | Description                                     |
| --------- | ------- | ----------------------------------------------- |
| `start`   | integer | Line number of the section heading (`### path`) |
| `fence`   | integer | Line number of the opening code fence           |
| `content` | integer | Line number of the first content line           |
| `length`  | integer | Number of content lines                         |

Position metadata enables AI assistants to navigate directly to a specific file's content using line ranges, without reading the entire archive. For example, an assistant can read lines `content` through `content + length - 1` to get exactly the file content.

#### `integrity` — Hash Verification

| Field           | Type   | Description                                                                      |
| --------------- | ------ | -------------------------------------------------------------------------------- |
| `manifest_hash` | string | Hash of the serialized `mdmeld:` YAML section (including the `mdmeld:` key line) |
| `content_hash`  | string | Hash of the content section (everything after the closing `---`)                 |

The hash algorithm is specified by `mdmeld.hash_algorithm`. Both values may be `"HASH_PLACEHOLDER"` in archives that have not been finalized.

### 3. Content Section

The content section follows the closing `---` of the frontmatter. Each file is represented as:

````markdown
### path/to/file.ext

```syntax
file content here
```
````

`````

Rules:
- Each file block starts with a level-3 heading (`### `) containing the file path
- A blank line separates the heading from the code fence
- The opening fence uses exactly `backtick_count` backticks, followed by the syntax hint
- File content follows on the next line, preserving exact content (no trailing newline added)
- The closing fence uses exactly `backtick_count` backticks on its own line
- A blank line separates each file block from the next

### Backtick Count

The `backtick_count` is the minimum number of backticks needed to avoid ambiguity with file content that itself contains backtick runs. The packer scans all text file contents for consecutive backtick sequences and chooses a count at least one greater than the longest run found, with a minimum of 4 and maximum of 8.

If any file content contains a backtick run of 8 or more, the packer must reject the file.

## Binary Files

Binary files (detected by byte distribution analysis) are base64-encoded before being placed in code fences. The `encoding` field is set to `"base64"` and `type` to `"binary"`. The `hash` is computed on the original bytes, not the base64 representation. The `size` is the original byte count.

## Text Detection

A file is classified as text if:
- It contains no null bytes (0x00)
- No more than 5% of bytes are non-text control characters (bytes 0x01–0x08, 0x0E–0x1F, excluding tab 0x09, newline 0x0A, carriage return 0x0D)
- At least 80% of bytes are printable ASCII (0x20–0x7E) or common whitespace

Empty files (0 bytes) are classified as text.

## Hashing

Two algorithms are supported:

- **xxh64** — xxHash 64-bit, output as 16-character lowercase hex string. Default. Fast, non-cryptographic, suitable for integrity checking.
- **sha256** — SHA-256, output as 64-character lowercase hex string. Available when cryptographic integrity is needed.

Hash inputs:
- **File hash**: computed on the raw file bytes (`Uint8Array`)
- **Content hash**: computed on the UTF-8 string of the content section
- **Manifest hash**: computed on the UTF-8 string of the `mdmeld:` YAML section (the serialized YAML text from `mdmeld:` through the last file entry, before `integrity:`)

## HASH_PLACEHOLDER

Archives generated by AI assistants or by hand may use the string `"HASH_PLACEHOLDER"` for any hash value. A finalize step computes the real hashes and replaces the placeholders. Unpackers should accept placeholder values without verification.

## File Ordering

Files in the manifest and content section appear in the same order. The recommended order is lexicographic by path (case-sensitive), but parsers must not depend on ordering.

## Syntax Mapping

The `syntax` field is derived from the file extension and is used as the info string on the code fence. Common mappings:

| Extension | Syntax |
|---|---|
| `.ts`, `.tsx` | `typescript` |
| `.js`, `.jsx` | `javascript` |
| `.py` | `python` |
| `.rs` | `rust` |
| `.go` | `go` |
| `.md` | `markdown` |
| `.json` | `json` |
| `.yml`, `.yaml` | `yaml` |
| `.css` | `css` |
| `.html` | `html` |
| `.sh` | `bash` |

Files with no recognized extension use `text` as the default syntax.

## Example

```markdown
<!-- MDMeld v1.0 archive — a directory tree encoded as markdown.
     Read the manifest below for file listing and positions.
     Spec: https://github.com/3leaps/mdmeld -->
---
mdmeld:
  version: "1.0.0"
  created: "2026-02-28T12:00:00Z"
  hash_algorithm: xxh64
  backtick_count: 4
  files:
    - path: README.md
      type: text
      size: 22
      hash: "a1b2c3d4e5f60718"
      syntax: markdown
      position:
        start: 20
        fence: 22
        content: 23
        length: 1
    - path: src/index.ts
      type: text
      size: 42
      hash: "b2c3d4e5f6071829"
      syntax: typescript
      position:
        start: 26
        fence: 28
        content: 29
        length: 3
integrity:
  manifest_hash: "c3d4e5f607182930"
  content_hash: "d4e5f60718293041"
---

### README.md

````markdown
# Hello from MDMeld
`````

### src/index.ts

```typescript
export function greet(name: string): string {
  return `Hello, ${name}!`;
}
```

```

The format follows semantic versioning. The `version` field in the manifest indicates which version of this specification the archive conforms to. Parsers should check the major version and reject archives with an unsupported major version.
```
